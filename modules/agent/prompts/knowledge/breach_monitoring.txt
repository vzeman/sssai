# Breach & Dark Web Monitoring — Testing Methodology

## Overview

Breach monitoring determines whether the target domain, its employees, or associated credentials
have appeared in publicly known data breaches or dark web dumps. This intelligence informs the
broader security posture assessment and prioritises defensive actions.

---

## Phase 0 Integration (Always Run for Security/Pentest Scans)

Run these tools at the START of every security scan before active testing:

### 1. Domain Breach Check (HIBP API)

```
breach_check(domain="example.com")
```

Returns: breach names, dates, data types, affected account counts.

### 2. Credential Leak Discovery

```
credential_leak_check(domain="example.com")
```

Checks common email patterns (admin@, info@, security@, contact@) against holehe and related tools
to identify services where domain emails are registered — indicating credential exposure surface.

### 3. Supplementary Shell Tools

```bash
# Check breach-parse datasets (if local breach data is available)
breach-parse @example.com /output/breach-parse.txt

# holehe — check if email is registered on 120+ services
holehe admin@example.com --only-used --output /output/holehe.txt

# theHarvester — gather emails from public sources for deeper checking
theHarvester -d example.com -b all -f /output/harvester.json
```

## Breach Findings — Severity Mapping

| Condition | Severity | Rationale |
|-----------|----------|-----------|
| Active credentials exposed (password + email combo) | Critical | Immediate account takeover risk |
| Plaintext or weakly-hashed passwords in breach | Critical | Direct credential risk |
| Recent breach (< 1 year) with passwords | High | Passwords likely still valid |
| Old breach (> 2 years) with passwords | Medium | Passwords may be rotated |
| Breach with PII only (names, addresses, phones) | Medium | Phishing/social engineering risk |
| Breach with email addresses only | Low | Spam/phishing surface |
| Breach with non-sensitive metadata | Info | Low direct risk |

## Scheduled Daily Monitoring

For clients with scheduled scanning enabled, breach monitoring should run daily:

1. Use the `breach_check` tool with the domain
2. Compare results against previous scan using `compare_results`
3. If `new_since_last_scan` is non-empty — trigger a notification immediately
4. Notification severity: `critical` if passwords exposed, `warning` if PII only

### Notification Trigger Logic

```python
# After breach_check completes:
if new_breaches:
    severity = "critical" if any password-related data_types else "warning"
    # The notification dispatcher will send Slack/Discord/email alerts
    # based on configured channels with min_severity threshold
```

## Remediation Guidance Template

For any breach findings, include these remediations:

**Immediate (if passwords exposed):**
- Force password resets for all accounts with @domain email addresses
- Revoke all active sessions / API tokens
- Enable MFA on all services
- Alert IT/security team for incident response

**Short-term:**
- Implement credential stuffing protection (rate limiting, CAPTCHA on login)
- Set up ongoing breach monitoring (daily scheduled scan)
- Review third-party service accounts registered with domain emails

**Long-term:**
- Adopt password manager policy for all staff
- Implement SSO to reduce credential sprawl
- Subscribe to HIBP Enterprise notifications for real-time alerts
- GDPR: Assess notification obligations (72-hour rule for processors)

---

## Compliance Implications

- **GDPR Article 33**: Data processors must notify supervisory authority within 72 hours of becoming aware of a breach
- **PCI-DSS Requirement 12.10**: Maintain an incident response plan; breaches may trigger PCI audit
- **HIPAA**: If PHI was exposed, breach notification rules apply
- **SOC 2**: Breach exposure affects availability and confidentiality trust service criteria
