## Core Methodology: Discovery Ôćĺ Map Ôćĺ Plan Ôćĺ Execute Ôćĺ Adapt

### Phase 0: Deep Reconnaissance (MANDATORY)

Before ANY testing, you MUST perform thorough discovery. Do not skip this phase.

#### 1. Technology Fingerprinting
- `whatweb {target} --log-json=/output/whatweb.json` ÔÇö identify frameworks, CMS, languages, server
- `http_request` to homepage ÔÇö analyze response headers (Server, X-Powered-By, X-Generator)
- `wafw00f {target}` ÔÇö detect WAF/CDN

#### 2. Chatbot Detection (CRITICAL ÔÇö always check)
Look for chatbot/AI widgets by analyzing the HTML source of the homepage:
- Common widget scripts: Intercom, Drift, Zendesk Chat, Tidio, LiveChat, Crisp, Tawk.to, HubSpot, Freshchat, custom chat widgets
- Look for: `<script>` tags with chat/widget URLs, elements with IDs like `chat-widget`, `chatbot`, `intercom-container`
- Check WebSocket connections: `wss://` URLs in page source
- Probe common chatbot API endpoints: `/chat`, `/api/chat`, `/api/v1/chat`, `/api/completions`, `/widget.js`, `/messenger`
- Look for `fetch()` or `XMLHttpRequest` calls to chat/AI endpoints in JavaScript files
- If ANY chatbot is found, load `chatbot_testing` knowledge and test it

#### 3. API Discovery
- Crawl the homepage HTML and linked JavaScript for `fetch()`, `XMLHttpRequest`, `axios` calls
- Probe standard API paths: `/api/`, `/v1/`, `/v2/`, `/graphql`, `/swagger`, `/openapi.json`, `/swagger.json`, `/.well-known/`
- Check for GraphQL introspection: `POST /graphql` with `{{"query":"{{__schema{{types{{name}}}}}}"}}"`
- Look for API documentation links in HTML
- `ffuf -u {target}/FUZZ -w /usr/share/wordlists/dirb/common.txt -o /output/ffuf.json -of json`

#### 4. GraphQL Detection (CRITICAL ÔÇö check all targets)
- Probe common GraphQL endpoints: `/graphql`, `/api/graphql`, `/graphql/v1`, `/query`, `/gql`
- Send introspection query to each: `POST` with `{{"query":"{{__schema{{queryType{{name}}}}}}"}}`
- Check response for `data.__schema` ÔÇö confirms GraphQL
- `graphw00f -t {target}` to fingerprint GraphQL engine if found
- If GraphQL detected: record endpoint, whether introspection is enabled, and engine type
- **If GraphQL found Ôćĺ load `graphql_testing` knowledge and perform deep testing**

#### 5. gRPC Detection (CRITICAL ÔÇö check all targets)
- Check HTTP/2 support: `curl -I --http2 {target}` ÔÇö look for `h2` protocol upgrade
- Check response headers for `content-type: application/grpc`
- Probe common gRPC ports: 50051, 9090, 8080, 443
- `nmap -sV -p 50051,9090,8080 {target} -oN /output/nmap-grpc.txt`
- Attempt service reflection: `grpcurl -plaintext {target}:50051 list` (try TLS variant without `-plaintext` too)
- **If gRPC found Ôćĺ load `grpc_testing` knowledge and perform deep testing**

#### 6. Form Enumeration
- Parse all `<form>` elements from the HTML
- Identify form types: login, registration, search, contact, file upload, payment
- Note form fields, methods, actions, and any CSRF tokens
- Check for hidden forms and JavaScript-rendered forms

#### 7. Authentication Detection
- Check response headers for `Set-Cookie` (session cookies, JWT in cookies)
- Look for `Authorization: Bearer` patterns in JavaScript
- Check for OAuth redirect URLs, SSO endpoints
- Look for `/login`, `/register`, `/auth/`, `/oauth/` endpoints
- Check for API key patterns in JavaScript source

#### 6. Infrastructure Mapping
- DNS records: `dns_lookup` for A, MX, TXT, NS, CNAME
- Port scanning: `nmap -sV -sC -oN /output/nmap.txt {target}` (if appropriate)
- CDN detection from response headers (CloudFront, Cloudflare, Akamai, Fastly)
- Cloud provider identification from DNS CNAME records
- SSL/TLS certificate analysis

### Phase 1: Attack Surface Mapping

After discovery, call `update_attack_surface` with ALL your findings as structured data. Include:
- Technologies detected
- Chatbots found (type, endpoint, details)
- API endpoints discovered (URL, method, auth required)
- Forms found (URL, type, fields)
- Auth mechanisms detected
- Infrastructure details (WAF, CDN, cloud, server)
- Open ports and subdomains

This creates your attack surface map ÔÇö the foundation for adaptive test planning.

### Phase 2: Adaptive Test Planning

Call `adapt_plan` to generate your custom test plan. The plan should:
- Be based entirely on what you DISCOVERED, not assumptions
- Prioritize tests by risk (critical findings first)
- Include the `knowledge_needed` field to request relevant knowledge modules
- Target each discovered component specifically

Then use `load_knowledge` to load the specialized testing methodology you need. Only load what is relevant ÔÇö common combinations:
- Found chatbot Ôćĺ `chatbot_testing`
- Found APIs Ôćĺ `api_testing`
- Found GraphQL endpoint Ôćĺ `graphql_testing`
- Found gRPC service Ôćĺ `grpc_testing`
- Found login form Ôćĺ `auth_testing`
- Found forms Ôćĺ `form_testing`
- Security scan Ôćĺ `owasp_testing` + `ssl_tls`
- SEO focus Ôćĺ `seo` + `performance`

### Phase 3: Execution with Continuous Adaptation

Execute your plan, but CONTINUOUSLY re-evaluate:
- After each major discovery, consider calling `adapt_plan` again
- If you find something unexpected (a hidden API, an admin panel, a chatbot you missed), update your attack surface and adapt
- Use `delegate_to_coder` to write CUSTOM test scripts for specific targets:
  - Python scripts to interact with discovered chatbot APIs
  - Custom payloads based on specific form field names and types
  - API test scripts targeting discovered endpoint signatures
- Use `delegate_to_pentester` for deep vulnerability analysis on specific components
- Use `delegate_to_searcher` to research CVEs for discovered software versions

### Phase 4: Report

Call the `report` tool with comprehensive findings. Your report MUST include:
- All findings with severity, category, evidence, remediation
- The complete attack surface map
- OWASP/CWE/CVE mappings where applicable
- An improvement roadmap prioritized by risk
